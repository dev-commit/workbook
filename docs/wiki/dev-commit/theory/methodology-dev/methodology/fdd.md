# Feature-Driven Development (FDD)

::: tip Feature-Driven Development

- **Feature-Driven Development** (FDD, разработка, управляемая функциональностью)
- Итеративная методология разработки ПО
- Фреймворк Agile для разработки ПО, который организует работу вокруг функций продукта
- _Особенность_: короткие итерации и частые релизы
- Project Business Owner (заказчик) должен присутствовать на первоначальном совещании по проектированию и ретроспективах итераций
  :::

## Виды деятельности в FDD

::: details 1. Разработка общей модели

**Описание**

- Общая модель создается главным архитектором или другим специалистом, руководящим проектом, путем определения области действия и контекста системы.

**Алгоритм**

1. Высокоуровневневый сквозной анализ широты решаемого круга задач и контекста системы
2. Для каждой моделируемой области делается более детальный сквозной анализ. Сквозные описания составляются в небольших группах и выносятся на дальнейшее обсуждение и экспертную оценку
3. Одна из предлагаемых моделей или их объединение становится моделью для конкретной области. Модели каждой области задач объединяются в общую итоговую модель, которая изменяется в ходе работы

:::

::: details 2. Составление списка необходимых функций системы

**Описание**

- Разработчики проводят мозговой штурм списка потенциальных элементов, которые будут полезны пользователям и могут быть завершены в установленные сроки для выпуска. Каждая функция должна быть управляемой в течение примерно двух недель

**Правила**

- Разбиение областей (domain) на предметные области (subject areas) с точки зрения функциональности
- Каждая отдельная подобласть соответствует какому-либо бизнес-процессу, шаги которого становятся списком функций (свойств). В данном случае функции - это маленькие части понимаемых пользователем функций, представленных в виде "действие - результат - объект", например, «проверка пароля пользователя»
- Разработка каждой функции должна занимать не более 2 недель, иначе задачу необходимо разбить на несколько подзадач, каждая из которых сможет быть завершена за установленный двухнедельный срок

:::

::: details 3. Планирование работы над каждой функцией

**Описание**

- Владение классами распределяется среди ведущих программистов путём упорядочивания и организации свойств (или наборов свойств) в классы
- Функции организованы по времени их создания и важности для клиента. Назначается право собственности на каждую функцию

:::

::: details 4. Проектирование функции

**Описание**

- Детали каждой функции производятся, проверяются и дорабатываются

**Алгоритм**

1. Для каждого свойства создается проектировочный пакет
2. Ведущий программист выделяет небольшую группу свойств для разработки в течение двух недель
3. Вместе с разработчиками соответствующего класса ведущий программист составляет подробные диаграммы последовательности для каждого свойства, уточняя общую модель
4. Пишутся «заготовки» классов и методов, и происходит критическое рассмотрение дизайна
   :::

::: details 5. Реализация функции

**Алгоритм**

1. После успешного рассмотрения дизайна данная видимая клиенту функциональность реализуется до состояния готовности
2. Для каждого класса пишется программный код
3. После модульного тестирования каждого блока и проверки кода завершенная функция включается в основной проект (build)

:::
